# 마크다운 프리뷰 렌더러 구현 계획

## 문서 목적

[설계 문서](/Users/anthony/Work/toyproject/markdown-preview/docs/superpowers/specs/2026-04-23-markdown-preview-renderer-design.md)를 실제 작업 순서로 분해한다. 이 문서는 "무엇을 먼저 만들고, 각 단계가 끝났다고 볼 기준이 무엇인지"를 고정하는 데 목적이 있다.

현재 저장소는 사실상 비어 있으므로, 1차 구현 계획은 다음 두 축으로 나눈다.

- 앱 골격을 세우는 초기 부트스트랩
- 설계 범위에 포함된 기능을 작은 단위로 순차 구현

## 구현 원칙

- 설계 문서의 범위를 넘는 기능은 넣지 않는다.
- Electron 보안 기본값을 유지한다.
- CLI와 GUI 책임을 분리한다.
- 렌더링 파이프라인은 순수 함수에 가깝게 유지해 테스트 가능성을 높인다.
- 자동 리로드와 Mermaid 오류는 "창 전체 실패"가 아니라 "국소 실패"로 처리한다.

## 기술 기준

현재 계획에서 고정하는 기술 기준은 다음과 같다.

- 구현 언어: `TypeScript`
- 패키지 매니저/런타임: `bun`
- 런타임 버전 관리: 프로젝트 루트의 `mise.toml`로 `bun` 버전 고정
- 린터/포맷터: `biome`
- 데스크톱 셸: `Electron`
- 마크다운 파서: `markdown-it`
- Mermaid 렌더링: 브라우저 컨텍스트에서 `mermaid.initialize({ startOnLoad: false, securityLevel: 'strict' })` 후 프로그램 방식 렌더
- Electron 보안 경계: `contextIsolation` 유지, `preload`를 통한 제한적 API 노출
- 파일 감시: 1차 구현은 Node 표준 API 기반으로 시작하고, 안정성 문제가 확인되면 별도 감시 라이브러리 도입을 후속 작업으로 분리

마지막 항목은 의도적인 선택이다. 현재 요구사항은 기능 구현이 우선이고, 저장소도 비어 있으므로 감시 라이브러리까지 초기에 늘리기보다 최소 의존성으로 출발하는 편이 낫다.

개발 명령 체계도 이에 맞춰 고정한다.

- 개발 도구 설치: `mise install`
- 의존성 설치: `bun install`
- 개발 실행: `bun run dev`
- 테스트 실행: `bun test`
- 정적 검증: `bun run check`
- 포맷 적용: `bun run format`

## 작업 단위 개요

구현은 6개 단계로 나눈다.

1. 프로젝트 부트스트랩
2. CLI 파일 선택 흐름 구현
3. Electron 창과 IPC 골격 구현
4. 마크다운 렌더링 파이프라인 구현
5. 자동 리로드와 에러 상태 구현
6. 테스트와 문서 정리

각 단계는 앞 단계를 기반으로 진행하되, 1단계 안에 짧은 기술 스파이크를 포함한다.

## 1단계: 프로젝트 부트스트랩

### 목적

빈 저장소를 "실행 가능한 최소 앱" 상태로 만든다.

### 작업

- `package.json` 생성
- `mise.toml` 생성
- `bun.lock` 생성
- `biome.json` 생성
- 실행 스크립트 정의
- Electron 메인 진입 파일 추가
- preload 파일 추가
- renderer용 정적 HTML/CSS/JS 파일 추가
- 기본 디렉토리 구조 확정

추천 구조는 다음과 같다.

```text
.
├── biome.json
├── bun.lock
├── mise.toml
├── package.json
├── src
│   ├── cli
│   │   ├── index.ts
│   │   ├── resolve-target.ts
│   │   ├── scan-markdown-files.ts
│   │   └── run-fzf.ts
│   ├── main
│   │   ├── index.ts
│   │   ├── create-window.ts
│   │   ├── ipc.ts
│   │   └── watch-file.ts
│   ├── preload
│   │   └── index.ts
│   ├── renderer
│   │   ├── index.html
│   │   ├── index.ts
│   │   ├── preview.css
│   │   ├── render-preview.ts
│   │   └── render-mermaid.ts
│   └── shared
│       ├── markdown
│       │   ├── render-markdown.ts
│       │   └── extract-mermaid-blocks.ts
│       └── types.ts
└── test
    ├── cli
    └── shared
```

초기 스크립트는 최소한 아래 수준으로 맞춘다.

- `dev`: Electron 개발 실행
- `check`: Biome 검사 실행
- `format`: Biome 포맷 실행
- `test`: Bun 테스트 실행

`mise.toml`에는 최소한 아래 내용을 반영한다.

- 프로젝트에서 사용하는 `bun` 버전을 명시적으로 고정
- 필요하면 `mise` 최소 버전도 함께 명시

이 항목은 선택이 아니라 필수다. 저장소에 버전 고정이 없으면 팀원별 `bun` 버전 차이로 설치와 실행 결과가 쉽게 갈린다.

### 기술 스파이크

이 단계에서 가장 먼저 확인할 것은 "보안 기본값을 유지한 상태로 renderer에서 Mermaid를 무리 없이 구동할 수 있는가"이다.

짧은 검증 항목:

- `BrowserWindow`가 `preload`와 함께 정상 기동하는가
- renderer가 로컬 HTML을 정상 로드하는가
- renderer 코드에서 Mermaid를 호출해 샘플 SVG를 그릴 수 있는가

이 스파이크가 통과하면 이후 단계의 구조를 그대로 유지한다. 실패하면 그때만 renderer 자산 번들링 계층을 추가한다. 즉, 번들러 도입은 기본 계획이 아니라 실패 시 대응책이다.

### 완료 기준

- `bun run dev` 단일 명령으로 빈 프리뷰 창이 열린다
- preload를 통한 제한적 API 노출 구조가 잡혀 있다
- Mermaid 샘플 다이어그램 렌더 가능 여부가 확인되어 기록된다
- `mise install` 후 동일한 `bun` 버전으로 환경이 재현된다
- `bun run check`와 `bun test`를 실행할 기본 토대가 잡혀 있다

## 2단계: CLI 파일 선택 흐름 구현

### 목적

사용자가 터미널에서 기대한 방식으로 파일을 선택하고 앱을 띄울 수 있게 한다.

### 작업

- 인자 0개/1개 규칙 구현
- 대상이 파일인지 디렉토리인지 판별
- 디렉토리 재귀 스캔 구현
- 제외 규칙 구현: `.git`, `node_modules`, 숨김 디렉토리
- `fzf` 설치 여부 확인
- `fzf` 실행 및 선택 결과 처리
- 취소 시 조용한 종료
- 오류 메시지 정리

### 세부 구현 방침

- 파일 스캔은 셸 파이프라인이 아니라 애플리케이션 코드로 구현한다
- `fzf`만 외부 명령으로 사용한다
- 경로 판별, 파일 수집, `fzf` 실행은 각각 별도 함수로 나눈다

이렇게 해야 테스트가 쉬워지고, 나중에 `fzf` 대신 다른 선택기를 붙이더라도 영향 범위가 작다.

### 완료 기준

- 인자 없음: `cwd` 기준으로 동작한다
- 파일 인자: 바로 대상 파일로 확정된다
- 디렉토리 인자: `*.md` 목록이 `fzf`에 표시된다
- `fzf` 취소 시 오류 없이 종료된다
- 오류 케이스 메시지가 문서화된다

## 3단계: Electron 창과 IPC 골격 구현

### 목적

CLI에서 결정된 파일 경로를 GUI 창으로 전달하고, 창 생명주기와 갱신 통로를 안정적으로 만든다.

### 작업

- CLI에서 선택된 파일 경로를 Electron 메인 프로세스로 전달
- 메인 프로세스에서 새 `BrowserWindow` 생성
- 창 제목에 파일명 반영
- preload에서 renderer로 노출할 최소 API 정의
- 초기 파일 데이터 전달용 IPC 정의
- 이후 갱신 이벤트 전달용 IPC 정의

### API 경계

1차 버전에서 renderer에 직접 노출할 기능은 최소화한다.

- 현재 파일 메타데이터 수신
- 렌더된 HTML 수신
- 파일 상태 변경 이벤트 수신
- 렌더러가 필요 시 Mermaid 후처리를 수행할 수 있는 최소 호출

경로 접근, 파일 읽기, 감시 시작과 중지는 모두 메인 프로세스 책임으로 둔다.

### 완료 기준

- CLI에서 파일 하나를 넘기면 새 창이 열린다
- renderer는 직접 파일 시스템 API에 접근하지 않는다
- 초기 로드와 후속 갱신이 같은 통로 규약을 따른다

## 4단계: 마크다운 렌더링 파이프라인 구현

### 목적

마크다운 원문을 VS Code와 유사한 HTML 프리뷰로 바꾼다.

### 작업

- `markdown-it` 인스턴스 초기화
- 기본 옵션 고정
- fenced code block 처리 구현
- `mermaid` fence를 일반 코드 블록과 분리하는 규칙 구현
- 렌더 결과를 preview 컨테이너에 주입
- 기본 프리뷰 CSS 작성

### 구현 방침

- raw HTML은 기본적으로 비활성화 상태에서 시작한다
- `markdown-it` 렌더링 함수는 순수 입력/출력 함수로 유지한다
- Mermaid 블록은 HTML 문자열에 직접 SVG를 섞어 넣기보다, 플레이스홀더 노드로 먼저 렌더한 뒤 renderer에서 후처리한다

이 구조의 장점은 두 가지다.

- 마크다운 렌더와 Mermaid 렌더 실패를 분리할 수 있다
- 재렌더 시 SVG 교체 로직이 단순해진다

### CSS 범위

1차 스타일 목표는 "VS Code와 거의 같은 화면"이 아니라 "VS Code 사용자에게 이질감이 적은 기본 프리뷰"다.

포함 범위:

- 본문 타이포그래피
- 제목/목록/인용문/테이블 기본 스타일
- 코드 블록과 인라인 코드 스타일
- Mermaid 블록 컨테이너와 에러 박스 스타일

제외 범위:

- 테마 전환
- 사용자 커스텀 CSS
- 이미지/링크 상대 경로 처리

### 완료 기준

- 대표 마크다운 샘플이 HTML로 안정적으로 렌더된다
- 일반 코드 블록과 Mermaid 블록이 구분 처리된다
- Mermaid 성공/실패가 블록 단위로 표시된다

## 5단계: 자동 리로드와 에러 상태 구현

### 목적

파일 변경을 감지하고 프리뷰만 부드럽게 갱신한다.

### 작업

- 현재 파일 감시 시작/중지 로직 구현
- 변경 이벤트 debounce 적용
- 파일 재읽기 후 재렌더링
- preview 컨테이너 부분 갱신
- 파일 삭제/접근 불가 상태 메시지 처리
- Mermaid 재렌더 흐름 정리

### 구현 방침

- 전체 창 새로고침은 피한다
- 읽기 실패와 파싱 실패를 구분한다
- 감시 대상 변경 또는 창 종료 시 watcher 정리 코드를 명시적으로 둔다

특히 watcher 정리는 필수다. 이 부분을 느슨하게 두면 다중 창 시나리오에서 중복 감시나 유령 이벤트가 발생하기 쉽다.

### 완료 기준

- 마크다운 파일 저장 후 프리뷰가 자동 갱신된다
- Mermaid 포함 문서도 재렌더 후 정상 갱신된다
- 파일 삭제 또는 일시 접근 실패 시 상태 메시지가 표시된다
- 창 종료 시 watcher가 해제된다

## 6단계: 테스트와 문서 정리

### 목적

핵심 동작을 회귀 가능하게 만들고, 이후 구현 확장을 위한 기준을 남긴다.

### 우선 테스트 대상

- CLI 인자 해석
- 파일/디렉토리 분기
- 재귀 스캔과 제외 규칙
- `fzf` 실행 래퍼의 입력/출력 처리
- 마크다운 렌더 함수
- Mermaid fence 검출
- watcher 이벤트가 재렌더 요청으로 이어지는지 확인

### 테스트 전략

- 순수 함수는 단위 테스트
- 파일 스캔과 watcher는 임시 디렉토리를 사용하는 통합 테스트
- Electron 전체 창 E2E는 1차 범위에서 제외
- 테스트 러너는 `bun test`를 기본으로 사용한다
- 코드 스타일 검증은 ESLint/Prettier 조합 대신 `Biome` 단일 체계로 유지한다

### 문서 작업

- README 초안 추가
- 설치 요구사항 명시: `mise`, `fzf`
- `mise install`, `bun install`, `bun run dev`, `bun test`, `bun run check` 사용 예시 추가
- `mise.toml`로 `bun` 버전을 고정한다는 점 명시
- 알려진 제약 명시: 상대 경로 링크/이미지 미지원

### 완료 기준

- 핵심 순수 함수 테스트가 통과한다
- 최소 사용법이 README에 정리된다
- 1차 범위와 비범위가 문서에서 명확히 구분된다
- Biome 기반 포맷/검사 명령이 문서와 실제 스크립트에서 일치한다

## 권장 작업 순서

실제 구현은 아래 순서를 권장한다.

1. 1단계 부트스트랩과 Mermaid 스파이크
2. 3단계 Electron 창/IPC 골격
3. 4단계 마크다운 렌더링 파이프라인
4. 2단계 CLI 파일 선택 흐름
5. 5단계 자동 리로드
6. 6단계 테스트와 README

이 순서를 추천하는 이유는 "창에 무엇인가를 안정적으로 그리는 것"이 먼저 검증되어야 CLI와 watcher가 붙었을 때 디버깅 축이 분리되기 때문이다.

## 체크포인트

중간 점검은 아래 세 지점에서 한다.

### 체크포인트 A

빈 창과 Mermaid 샘플 렌더가 뜨는 시점

확인 질문:

- 보안 기본값을 유지한 상태에서 renderer 자산 구성이 충분히 단순한가
- 추가 번들러가 정말 필요한가

### 체크포인트 B

마크다운과 Mermaid 혼합 문서가 한 번 정상 렌더되는 시점

확인 질문:

- Mermaid 플레이스홀더 구조가 재렌더에 적합한가
- CSS 방향이 과도하게 커지고 있지 않은가

### 체크포인트 C

자동 리로드와 오류 표시가 붙는 시점

확인 질문:

- watcher 생명주기 관리가 단순한가

## 리스크와 대응

### 1. Mermaid 로딩 방식이 예상보다 까다로울 수 있음

대응:

- 1단계에서 먼저 기술 스파이크로 검증한다
- 실패 시에만 renderer 자산 번들링을 추가한다

### 2. 파일 감시가 플랫폼별로 다르게 동작할 수 있음

대응:

- 1차 구현은 로컬 개발 환경에서 가장 단순한 방식으로 시작한다
- 이벤트 누락, 중복, rename 처리 문제가 보이면 감시 계층만 교체 가능하도록 감싼다

### 3. VS Code와 "완전히 같은" 시각 결과를 기대하게 될 수 있음

대응:

- 현재 목표는 출력 호환성과 사용감 유사성이다
- 상대 경로 처리, extension 호환, VS Code 내부 CSS 복제는 범위 밖으로 유지한다

## 구현 완료 정의

아래 조건을 모두 만족하면 1차 구현 완료로 본다.

- 터미널 명령으로 파일 또는 디렉토리를 입력할 수 있다
- 디렉토리 입력 시 `fzf`로 `*.md` 파일을 선택할 수 있다
- 선택된 파일이 새 GUI 창에서 렌더된다
- 기본 Markdown/GFM 성격 문서가 읽기 좋은 형태로 표시된다
- `mermaid` 코드 블록이 SVG로 렌더된다
- 파일 저장 시 프리뷰가 자동 갱신된다
- 오류가 창 전체 실패가 아니라 국소 상태로 표시된다
- README에 설치 및 실행 방법이 정리되어 있다
