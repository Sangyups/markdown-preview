# 마크다운 프리뷰 렌더러 설계

## 개요

터미널에서 실행하는 CLI 중심의 마크다운 프리뷰 앱을 만든다.

- 인자가 없으면 현재 작업 디렉토리를 대상으로 삼는다.
- 인자가 파일이면 해당 파일을 바로 연다.
- 인자가 디렉토리면 `fzf`로 `*.md` 파일을 선택한다.
- 파일이 정해지면 새로운 GUI 창을 열어 마크다운을 렌더한다.
- 창이 열린 뒤에는 파일 변경을 감지해 자동으로 다시 렌더한다.

1차 버전의 구현 기반은 `Electron`으로 잡는다. 이유는 독립 GUI 창을 가장 빠르게 안정적으로 제공할 수 있고, Chromium 기반 렌더링 환경 덕분에 VS Code의 기본 마크다운 프리뷰와 유사한 출력 결과를 맞추기 쉽기 때문이다.

## 목표

- VS Code의 기본 마크다운 프리뷰와 최대한 비슷한 렌더 결과를 낸다.
- 기본 Markdown/GFM 스타일 렌더링을 지원한다.
- `mermaid` 코드 블록을 기본 지원한다.
- 터미널에서 예측 가능한 방식으로 파일을 선택하고 GUI 프리뷰를 연다.
- 프리뷰 창은 `preview-only`로 유지한다.
- 대상 파일 변경 시 자동 리로드한다.

## 비목표

- VS Code 내부 프리뷰 구현을 그대로 재사용하는 것
- VS Code extension 호환 구조를 처음부터 제공하는 것
- 상대 경로 링크 및 이미지 해석
- split editor/preview UI
- 최근 파일 목록, 탭 UI, 세션 복원 같은 부가 기능

## 사용자 동작 규칙

### CLI 진입 규칙

명령은 인자를 0개 또는 1개 받는 것을 기본으로 한다.

- 인자가 없으면 `cwd`를 대상 경로로 사용한다.
- 인자가 파일이면 해당 파일을 바로 연다.
- 인자가 디렉토리면 하위에서 `*.md`를 재귀적으로 찾은 뒤 `fzf`로 선택하게 한다.

디렉토리 스캔 시 다음 경로는 기본 제외한다.

- `.git`
- `node_modules`
- 숨김 디렉토리

사용자가 `fzf` 선택을 취소하면 조용히 종료한다. `fzf`가 설치되어 있지 않으면 CLI에서 설치 필요 메시지를 출력하고 종료한다.

### GUI 창 규칙

GUI는 `preview-only` 창이다. 편집 기능은 넣지 않는다. 파일 하나를 선택할 때마다 새 창 하나를 연다. 창 제목에는 파일명을 포함해 여러 창이 떠 있어도 구분 가능하게 한다.

GUI 창의 책임은 다음 세 가지로 제한한다.

- 선택된 파일 읽기
- 마크다운을 HTML로 렌더하기
- 파일 변경을 감시하고 프리뷰 갱신하기

파일 선택과 경로 판별은 CLI가 맡고, 표시와 갱신은 GUI가 맡는 구조로 분리한다.

## 아키텍처

구조는 네 개의 단위로 나눈다.

### 1. CLI 런처

대상 경로를 해석하고, 디렉토리면 파일 목록을 만들고, `fzf`를 실행해 최종 마크다운 파일 경로를 결정한다. 결정된 파일 경로를 Electron 앱 프로세스에 전달한다.

### 2. Electron 메인 프로세스

`BrowserWindow`를 생성하고, 선택된 파일 경로를 받아 윈도우 생명주기를 관리한다. 파일 변경 감시를 설정하고, 렌더러 프로세스에 필요한 데이터 갱신을 전달한다.

### 3. Renderer 프로세스

실제 프리뷰 화면을 그린다. 렌더된 HTML을 표시하고, 브라우저 환경에서 `mermaid`를 SVG로 변환한다. 마크다운 또는 다이어그램 단위 에러를 화면에 국소적으로 표시한다.

### 4. 마크다운 렌더링 파이프라인

마크다운 문자열을 HTML로 바꾸는 순수 렌더링 계층이다. VS Code 내부 API에 직접 의존하지 않고, `markdown-it` 기반 파이프라인으로 VS Code 기본 프리뷰와 유사한 출력을 목표로 한다.

## 렌더링 파이프라인

### 마크다운 렌더링

핵심 전략은 "VS Code 내부 구현 재사용"이 아니라 "출력 결과 호환성에 집중한 독립 파이프라인"이다.

기본 범위는 다음과 같다.

- CommonMark 계열 기본 문법
- fenced code block
- 일반적인 GFM 성격의 편의 문법
- 코드 블록 syntax highlighting

이 방식은 독립 앱으로 패키징하기 쉽고, VS Code 전용 webview 제약에 묶이지 않는다.

### Mermaid 렌더링

`mermaid`는 1차 범위에 포함한다.

- 언어 태그가 `mermaid`인 fenced code block을 별도 블록으로 인식한다.
- 마크다운 렌더 결과에 Mermaid 대상 노드를 배치한다.
- 브라우저 컨텍스트에서 `mermaid.initialize()`를 한 번 수행한다.
- 각 Mermaid 블록을 SVG로 렌더해 삽입한다.

Mermaid 렌더 실패는 전체 창을 깨지 않고 해당 블록만 에러 상태로 대체한다. 보안 설정은 느슨한 모드보다 보수적인 쪽에서 시작한다.

### 자동 리로드

파일이 변경되면 다음 순서로 처리한다.

1. 파일을 다시 읽는다.
2. 마크다운을 다시 HTML로 변환한다.
3. 프리뷰 컨테이너를 갱신한다.
4. 새 컨텐츠 기준으로 Mermaid 렌더링을 다시 수행한다.

전체 윈도우 새로고침보다 프리뷰 컨텐츠만 교체하는 방식을 우선한다. 이렇게 해야 깜빡임이 적고 리로드 체감이 자연스럽다.

## 에러 처리

실패는 단순하고 명확하게 드러내야 한다.

### CLI 에러

- 대상 경로가 존재하지 않음
- 대상 파일을 읽을 수 없음
- 디렉토리 안에 `*.md`가 없음
- `fzf`가 설치되어 있지 않음

이 경우 CLI에서 이해 가능한 메시지를 출력하고 종료한다.

### 프리뷰 에러

- 마크다운 파싱 실패: 프리뷰 영역에 에러 박스 표시
- Mermaid 파싱 또는 렌더 실패: 해당 다이어그램 블록만 에러 표시
- 감시 중인 파일이 삭제되거나 일시적으로 접근 불가: 상태 메시지 표시

컨텐츠 단위 오류 때문에 전체 창이 종료되면 안 된다.

## 테스트 전략

1차 버전은 GUI E2E보다 단위 테스트와 얕은 통합 테스트를 우선한다.

### 테스트 대상

- CLI 인자 해석
- 파일/디렉토리 분기 처리
- 제외 규칙을 포함한 재귀 디렉토리 스캔
- `fzf` 호출 구성
- 대표 마크다운 입력에 대한 HTML 렌더 결과
- `mermaid` fence 검출과 변환
- 파일 변경 감지 후 재렌더 트리거

### 후순위 테스트

- 데스크톱 전체 E2E 자동화
- 플랫폼별 패키징 검증
- 상대 경로 링크/이미지 처리

## Electron 선택 이유

이번 요구사항의 우선순위는 두 가지다.

- 독립 GUI 창을 빠르게 제공할 것
- VS Code 프리뷰와 유사한 렌더 결과를 낼 것

`Electron`은 이 두 가지에 가장 직접적으로 맞는다. `Tauri`는 더 가볍지만 초기 구현 복잡도가 올라간다. 로컬 서버 + 브라우저 방식은 더 단순할 수 있지만, 사용자가 원하는 "파일 선택 후 별도 GUI 창이 열리는 앱"이라는 제품 형태와는 거리가 있다.

## 향후 확장 포인트

현재 구조는 다음 기능을 나중에 추가할 수 있도록 열어 둔다.

- 상대 경로 링크 및 이미지 처리
- split editor/preview 모드
- 테마 및 스타일 커스터마이징
- 파일 글롭 패턴 옵션화
- 창 재사용 또는 다중 파일 탐색

다만 이 기능들은 모두 1차 구현 범위에서 제외한다.
